# godot-cli-control Skill 与 init 集成设计

- 日期：2026-04-29
- 作者：kesar（与 Claude 协作头脑风暴）
- 状态：草稿，待评审

## 目标

为 `godot-cli-control` 增加一份"agent 使用说明"形式的 skill，并改造 `godot-cli-control init` 子命令，让它在用户的 Godot 项目里同时落盘 Claude Code 与 Codex 两份 skill 文件。任何在该项目中工作的 AI agent 因此能立即知道这套 CLI 的存在、调用方式和适用边界，无需用户每次手把手介绍。

## 非目标

- 不为 Claude / Codex 各写差异化 skill 内容（共用同一份模板源即可）
- 不引入模板引擎（Jinja 等），仅做 `str.replace` 占位符替换
- 不写 SKILL.md 内容质量的自动化断言（人评 + reviewer agent 兜底）
- 不端到端验证 agent 真的会读取 skill（超出本次范围）
- 不改 `AGENTS.md` / `CLAUDE.md`（用户已选择独立 skill 文件路径）

## 背景

`godot-cli-control` 是一个 Godot 4 插件 + Python CLI / `GameClient` 的组合，通过 WebSocket 桥让外部进程可以驱动一个跑着的 Godot 场景：点击节点、读写属性、模拟输入动作、截图、录像、转储场景树。命令面包含 `daemon start/stop`、`init`、`run <script>` 以及一组 RPC 子命令（`click / screenshot / tree / press / release / tap / hold / combo / release-all`）；Python 侧提供 `GameClient` async API（约 18 个方法）。

现有 `init` 子命令已经能在用户的 Godot 项目根：

1. 校验 `project.godot` 存在
2. 复制 `addons/godot_cli_control/`
3. patch `project.godot`（autoload + editor_plugins）
4. 检测并写入 `.cli_control/godot_bin`

本次新增第 5 步：写入 Claude / Codex 的 skill 文件。

## 用户决策汇总

头脑风暴中已落锤的四点（按问答顺序）：

1. **"通过 init"指的是 `godot-cli-control init`**，不是 Claude 的 `/init` skill。
2. **落盘形式**：两边都走独立 skill 文件，不动 `AGENTS.md` / `CLAUDE.md`：
   - `<project_root>/.claude/skills/godot-cli-control/SKILL.md`
   - `<project_root>/.codex/skills/godot-cli-control/SKILL.md`
3. **源头组织**：单一 `SKILL.md` 模板，Claude / Codex 共用同一份内容。
4. **幂等策略**：默认覆盖（保证版本跟随），`--no-skills` 跳过；新增 `--skills-only` 仅写 skill、跳过插件复制 / project.godot patch / godot_bin 检测。
5. **内容范围**：覆盖 CLI、Python `GameClient` API、`def run(bridge)` 脚本三种使用面，并给出 trigger 指引。
6. **方案**：模板 + 占位符（B 方案），版本号与 CLI 帮助文本由 `init` 渲染时注入，其余正文为手写。

## 架构总览

两条产物线（Claude `.claude/skills/...`、Codex `.codex/skills/...`），一份模板源：

```
godot-cli-control 仓库
└── python/godot_cli_control/
    ├── templates/skill/
    │   └── SKILL.md          ← 唯一模板源（含 {{cli_help}} {{version}} 占位符）
    ├── skills_install.py     ← 新增：渲染+落盘的纯函数模块（可独立单测）
    ├── init_cmd.py           ← 调用 skills_install.install_skills(...)
    └── cli.py                ← init 子命令新增 --no-skills / --skills-only
                              ← 暴露 build_parser() 给 skills_install 取 --help
```

`godot-cli-control init` 完整流程：

1. 校验 `project.godot` 存在
2. 复制插件到 `addons/godot_cli_control/`
3. patch `project.godot`
4. 检测 Godot 二进制写 `.cli_control/godot_bin`
5. **新增**：`skills_install.install_skills(project_root, version=..., cli_help=..., force=True)`
   - 渲染模板（替换 `{{version}}`、`{{cli_help}}`）
   - 写入 `<project_root>/.claude/skills/godot-cli-control/SKILL.md`
   - 写入 `<project_root>/.codex/skills/godot-cli-control/SKILL.md`
   - `--no-skills` 跳过本步；`--skills-only` 跳过 1-4，只跑本步

## SKILL.md 模板内容

### Frontmatter

```yaml
---
name: godot-cli-control
description: Use when driving a Godot 4 game from a script or terminal — clicking buttons, simulating input actions, taking screenshots, dumping the scene tree, or recording demos. Trigger when the user mentions godot-cli-control, the godot-cli-control CLI/daemon, or asks to automate / scrape / black-box-test a Godot scene.
---
```

`description` 同时作为 Claude superpowers 与 Codex skills 的 trigger 索引来源。

### 正文骨架（目标 ~250 行）

1. **What this is**（1 段）— WebSocket bridge：daemon 进程托管 Godot；agent 通过 CLI 或 Python `GameClient` 发 RPC。
2. **When to use** —
   - 用：UI 黑盒测试、回归录屏、写 demo bot、自动化截图、脚本化输入
   - 不用：常规 GDScript 单测（让 GUT 干）、需要直接读源码逻辑的场景
3. **Quick start** — `init → daemon start → tree 3` 三步最短路径
4. **CLI 速查** — 占位符段落，由 init 渲染时注入：
   ```
   <!-- BEGIN cli_help (auto-generated by godot-cli-control init) -->
   {{cli_help}}
   <!-- END cli_help -->
   ```
5. **Python `GameClient` API** — 手写表，每个方法一行签名 + 一句用途。覆盖：`click / get_property / set_property / call_method / get_text / node_exists / is_visible / get_children / screenshot / get_scene_tree / wait_for_node / wait_game_time / action_press / action_release / action_tap / hold / combo / combo_cancel / release_all`。
6. **`def run(bridge)` 脚本模式** — 一段最小可跑示例 + `godot-cli-control run my_script.py` 调用方式；说明 `bridge` 与 `GameClient` 的关系。
7. **常见坑（短）**：
   - daemon 必须先 `start`，否则所有 RPC 立刻 `ConnectionRefused`
   - 节点路径必须绝对（`/root/...`）
   - `all_proxy` 等环境变量会劫持 localhost — 让用户 `unset` 或忽略；client.py 已显式 `proxy=None` 兜底
8. **Footer**：`Generated from godot-cli-control v{{version}}. Re-run `godot-cli-control init --skills-only` to refresh.`

## `skills_install.py` 接口

```python
# python/godot_cli_control/skills_install.py
from __future__ import annotations
from importlib.resources import files
from pathlib import Path

CLAUDE_REL = Path(".claude/skills/godot-cli-control/SKILL.md")
CODEX_REL  = Path(".codex/skills/godot-cli-control/SKILL.md")


def render_skill(version: str, cli_help: str) -> str:
    """读模板 → 占位符替换 → 返回最终文本。纯函数，无 IO 副作用。"""
    template = (files("godot_cli_control.templates.skill") / "SKILL.md").read_text(encoding="utf-8")
    return (
        template
        .replace("{{version}}", version)
        .replace("{{cli_help}}", cli_help)
    )


def install_skills(
    project_root: Path,
    *,
    version: str,
    cli_help: str,
    force: bool = True,
) -> list[Path]:
    """渲染并写入两条目标路径。返回实际写入的路径列表。

    force=True：覆盖；force=False 且文件存在：跳过该路径。
    父目录按需 mkdir(parents=True, exist_ok=True)。
    """
    content = render_skill(version, cli_help)
    written: list[Path] = []
    for rel in (CLAUDE_REL, CODEX_REL):
        dst = project_root / rel
        if dst.exists() and not force:
            continue
        dst.parent.mkdir(parents=True, exist_ok=True)
        dst.write_text(content, encoding="utf-8")
        written.append(dst)
    return written
```

## 调用侧改动

### `init_cmd.run_init`

```python
def run_init(project_root, force=False, install_skills_=True, skills_only=False):
    if not skills_only:
        # 原 1-4 步保持不动
        ...

    if install_skills_:
        from . import skills_install, cli, _version
        help_text = cli.build_parser().format_help()
        version = getattr(_version, "__version__", "unknown")
        written = skills_install.install_skills(
            project_root,
            version=version,
            cli_help=help_text,
            force=True,  # init 默认即覆盖
        )
        for p in written:
            print(f"写入 skill：{p.relative_to(project_root)}")
```

### `cli.py`

```python
init_p.add_argument("--no-skills", action="store_true",
                    help="跳过 .claude/.codex skill 写入")
init_p.add_argument("--skills-only", action="store_true",
                    help="只写 skill 文件，跳过插件复制 / project.godot patch / godot_bin 检测")

def cmd_init(ns):
    return run_init(
        project_root=Path(ns.path or "."),
        force=ns.force,
        install_skills_=not ns.no_skills,
        skills_only=ns.skills_only,
    )

# 必须暴露 build_parser 供 skills_install 取 --help（之前是私有 _build_parser）
def build_parser() -> argparse.ArgumentParser: ...
```

### 循环依赖防护

`skills_install.py` 不 import `cli.py`，由 `init_cmd` 充当胶水从 `cli.build_parser()` 取 help、`_version` 取版本，再注入到 `install_skills(..., cli_help=, version=)`。`skills_install` 自身只依赖 `importlib.resources` + `pathlib`，可独立单测且零外部副作用。

### 打包

`pyproject.toml` 确保 `templates/` 目录入 wheel：

```toml
[tool.hatch.build.targets.wheel.force-include]
"../addons/godot_cli_control" = "godot_cli_control/_plugin"
"godot_cli_control/templates" = "godot_cli_control/templates"  # 新增（如默认 include 已带上则可省）
```

实际是否需要显式 force-include 以打包验证为准（hatch 默认会带 package 内目录，但如果 templates 不含 `__init__.py` 可能被跳过）。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| `templates/` 不含 `__init__.py`，被 hatch 跳过 | 打包后立即跑 `python -c "from importlib.resources import files; print(files('godot_cli_control.templates.skill') / 'SKILL.md')"` 验证；必要时加 force-include |
| 用户在 `.claude/skills/godot-cli-control/SKILL.md` 自定义后被 init 静默覆盖 | 默认覆盖是用户主动决定（问答 4），但在 README "Agent integration" 段明确警告：「自定义请放到不同 skill 名下，或加 `--no-skills` 后手动维护」 |
| Codex 后续协议偏离 Claude，单一模板出现分歧 | 现阶段单源；真出现时把 `install_skills` 改成读两份模板，是 < 30 行的局部改动 |
| `cli.build_parser().format_help()` 输出含本机绝对路径 | 检查 argparse `prog` 设置，必要时 init 渲染前对 help 文本做 prog 归一化（`sys.argv[0]` → `godot-cli-control`） |
